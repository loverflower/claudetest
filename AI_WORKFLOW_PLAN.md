# План обучающего проекта: Todo-приложение через 8 шагов AI-мануала

## 1. Введение

Цель упражнения — не «написать todo-лист», а **отработать на практике все 8 шагов** из `AI manual for individual exploration.md`, используя todo-приложение как полигон. Стек зафиксирован в SCRUM-5: **React + TypeScript + Tailwind CSS**, без backend.

8 шагов мануала (краткое напоминание):

| # | Шаг | Что отрабатываем |
|---|-----|------------------|
| 1 | Project map (Cartograph) | AI-навигация по репозиторию |
| 2 | MCP setup | Получение задач без выхода из IDE |
| 3 | Refine task (`/dor`) | Definition of Ready — превращение тикета в спеку для агента |
| 4 | Plan (Plan Mode / `/plan`) | Read-only планирование с мощной моделью |
| 5 | Validate plan | Привычка ревьюить план до approve |
| 6 | Test | End-to-end проверка фичи |
| 7 | Auto review | Машинное ревью до человека (CodeRabbit / `/ultrareview`) |
| 8 | Manual review | Ответственность остаётся на человеке |

Шаги 1 и 2 — одноразовые (setup). Шаги 3–6 циклически повторяются для каждого тикета. Шаги 7–8 — на каждый PR.

## 3. Действия по каждому шагу мануала в нашем проекте

### Шаг 1. Project map (Cartograph)
- **Что делаем:** добавляем Skill `Cartograph` в `.claude/skills/Cartograph/SKILL.md`, запускаем `/cartographer`, результат подключаем строкой в `CLAUDE.md`.
- **Когда:** **откладываем** до завершения SCRUM-5 (см. отдельный раздел ниже).
- **Артефакт:** `docs/CODEBASE_MAP.md` + ссылка из `CLAUDE.md`.

### Шаг 2. MCP setup
- **Статус:** выполнен. Atlassian Rovo подключён, тикеты SCRUM-3/5/6/7/8 видны.
- **Дополнительно:** проверить, что доступны три ключевые операции: search (JQL), get issue, get comments. На тикете SCRUM-5 прогнать `searchJiraIssuesUsingJql` + `getJiraIssue` как дымовой тест.
- **Артефакт:** пометка в `CLAUDE.md` о подключённом MCP (одна строка).

### Шаг 3. Refine task — `/dor`
- **Что делаем:** для каждого тикета (начиная с SCRUM-5) запускаем skill `/dor`, отвечаем на уточняющие вопросы агента, сохраняем результирующую спецификацию.
- **Инструменты:** Skill `dor`, MCP `getJiraIssue` для подгрузки описания.
- **Артефакт:** Markdown-спека на тикет (objective, scope/non-scope, constraints, files, acceptance criteria, ссылка на Jira). Можно держать в `docs/specs/SCRUM-XX.md`.

### Шаг 4. Planning
- **Что делаем:** в Claude Code переходим в **Plan Mode** (`Shift+Tab` до режима plan) или используем `/plan` со спекой из шага 3 в качестве входа. Используем сильную модель (Opus).
- **Артефакт:** план реализации в чате — список файлов, изменений, последовательности шагов, тест-кейсы.

### Шаг 5. Validate plan
- **Что делаем:** перед approve вычитываем ≥ 80% плана. Чек-лист валидации:
  - нет ли overengineering (преждевременные абстракции, лишние библиотеки),
  - не уходит ли план за рамки acceptance criteria,
  - покрывает ли все edge-cases из спеки,
  - совпадает ли список файлов с реальной структурой репо.
- **После approve:** переключаемся на более быструю модель (Sonnet) для имплементации.
- **Артефакт:** approved plan + смена модели.

### Шаг 6. Implement & test
- **Что делаем:** даём агенту выполнить план. Затем локально: `npm run dev`, прокликиваем acceptance criteria, прогоняем `npm run lint` / `tsc --noEmit` / unit-тесты, если они добавлены. При ошибке — цикл «опиши ошибку → агент чинит → ретест».
- **Артефакт:** работающая фича на dev-сервере + зелёный TypeScript/линт.

### Шаг 7. Auto review
- **Что делаем:** на каждом PR — машинное ревью. См. отдельный раздел «Auto review».
- **Артефакт:** комментарии бота в PR + наши решения «fix / wontfix / out of scope».

### Шаг 8. Manual review
- **Что делаем:** ручное ревью PR с теми же стандартами, что в боевых проектах. См. раздел «Manual review».
- **Артефакт:** approve и merge в `main`.

## 4. Мини-чеклист по каждому тикету

Применяем индивидуально к SCRUM-5, SCRUM-8, SCRUM-6, SCRUM-7:

1. **Refine** — `/dor` + ответы на уточнения, спека готова.
2. **Plan** — Plan Mode со спекой, получаем дерево изменений.
3. **Validate** — ревью плана по чеклисту из шага 5.
4. **Implement** — Sonnet выполняет план, переключаем модель.
5. **Test** — локальный прогон acceptance criteria + lint + tsc.
6. **Commit** — атомарный коммит/ветка `feature/SCRUM-XX-...`, сообщение со ссылкой на тикет.
7. **Push & PR** — пушим ветку на GitHub **только после явного «ок» пользователя**, затем `gh pr create` в `main` с линком на Jira и acceptance criteria.
8. **CI** — дождаться зелёного workflow `build` (см. §6.2). Если красный — чиним до auto review.
9. **Auto review** — запускаем CodeRabbit (на PR) и/или `/ultrareview` локально, отрабатываем замечания.
10. **Manual review** — самоконтроль по чеклисту (раздел ниже), затем merge.
11. **Jira transition** — через MCP (`transitionJiraIssue`) переводим тикет в Done.

## 5. Когда делаем Cartograph

**Откладываем шаг 1 до момента после завершения SCRUM-5.** Обоснование:

- Сейчас в репо только три файла (README, main.txt, manual). Cartograph по пустому репо выдаст бесполезную карту.
- SCRUM-5 создаёт реальную структуру (`src/components`, `src/hooks`, `src/types`, конфиги Vite/Tailwind/TS). Только после этого карта приобретает ценность.
- Триггер запуска `/cartographer`: первый зелёный `npm run dev` после SCRUM-5, до начала работы над SCRUM-8.
- После SCRUM-7 (последний тикет) — повторный прогон Cartograph, чтобы карта отражала финальное состояние. Это и тренировка инкрементального обновления карты.

Промежуточный компромисс на старте: пока карты нет, поддерживаем минимальный `CLAUDE.md` с фиксированными правилами (стек, naming, расположение хуков), чтобы агент не блуждал.

## 6. GitHub: push policy и CI/CD

### 6.1. Push policy

- **Главное правило: никогда не коммитить и не пушить напрямую в `main`.** Все изменения идут через feature/chore/fix-ветку → push → PR → approve пользователя → merge.
- Локально: коммитим только в feature-ветку. Если активная ветка `main` — сначала создать новую ветку, потом коммитить.
- Push на `origin`: **только после явного «ок, пушим» от пользователя.**
- После push — `gh pr create` (с одобрения) в `main`. В описании PR: ссылка на Jira-тикет, acceptance criteria из спеки, какие команды verification прошли.
- Merge — пользователь сам через GitHub UI. Я НЕ выполняю `gh pr merge`.
- Force-push в `main` — категорически запрещён. Force-push в feature-ветку — только по явной просьбе пользователя.

### 6.2. Учебный CI/CD workflow

Цель — простейший GitHub Actions workflow для отработки практики, не «энтерпрайз-сборка».

- **Файл:** `.github/workflows/ci.yml`.
- **Триггеры:** `push` в `main` и `pull_request` в `main`.
- **Job `build`** (single job, `ubuntu-latest`):
  1. `actions/checkout@v4`
  2. `actions/setup-node@v4` (Node 20, `cache: npm`)
  3. `npm ci`
  4. `npm run lint` (ESLint — если в SCRUM-5 решим ставить; иначе шаг убираем до того момента)
  5. `npx tsc --noEmit` (type check)
  6. `npm run build` (vite build)
- **Опционально (если в проекте появятся unit-тесты):** шаг `npm test -- --run`.
- **Когда заводим:** одновременно с первым push после SCRUM-5, чтобы PR по SCRUM-5 сразу шёл через зелёный CI. Раньше — нечего собирать; позже — теряется смысл первого ревью с CI.
- **Бэйдж в README:** после первого зелёного прогона добавить `![CI](https://github.com/<owner>/<repo>/actions/workflows/ci.yml/badge.svg)` — это часть учебной задачи.
- **Branch protection (опционально):** требовать прохождение `build` для merge в `main`. За рамками MVP мануала, делаем только при желании.

### 6.3. Что нужно от пользователя для GitHub-части

- Имя GitHub-репозитория (видимо `claudetest` под твоим аккаунтом) и подтверждение, что remote `origin` уже настроен (проверим перед первым push).
- Подтверждение «ок, ставим CI вместе с SCRUM-5» — иначе CI может уехать в отдельный under-the-radar тикет.
- Решение по CodeRabbit (см. §7) — нужен ли GitHub App или ограничиваемся `/ultrareview`.

## 7. Auto review

**Рекомендация:** на учебном проекте использовать `/ultrareview` локально (быстрая обратная связь, ноль настройки) **плюс** один раз настроить CodeRabbit на репозиторий — это сам по себе пункт из мануала.

| Инструмент | Когда | Зачем |
|------------|-------|-------|
| `/ultrareview` | Локально, сразу после `git commit`, до `git push` | Поймать очевидные проблемы до публичного PR, дешёвый цикл |
| CodeRabbit (GitHub App) | Автоматически на каждый PR | Полноценная отработка шага 7 мануала, имитация командного процесса |

Что ожидаем от auto review:
1. саммари изменений,
2. риски/регрессии,
3. пробелы в тестах и валидации,
4. неочевидные edge-cases.

Реакция: применяем здравый смысл, не чиним всё подряд. Замечания вне scope тикета помечаем «out of scope» и при необходимости заводим follow-up.

## 8. Manual review

Финальное ручное ревью каждого PR — по чек-листу:

- **Соответствие acceptance criteria** из спеки `/dor`. Все пункты выполнены, лишнего нет.
- **TypeScript strict** — нет `any`, нет `@ts-ignore` без обоснования, типы экспортируются из `src/types`.
- **Декомпозиция** — компоненты в `src/components`, бизнес-логика в `src/hooks` (требование SCRUM-6 — кастомный хук).
- **Tailwind** — без инлайн-стилей, без CSS-файлов помимо tailwind-входа; нет «магических» значений вне theme.
- **Состояние** — единый источник правды (один владелец списка todo), нет дублирующего state.
- **Persistence (после SCRUM-8)** — корректная сериализация/десериализация, защита от мусора в localStorage (try/catch + валидация формы данных).
- **A11y базово** — labels у инпутов, focus states, semantic HTML.
- **Чистота истории** — атомарные коммиты, сообщение `SCRUM-XX: ...`.
- **README** — обновлён под фактическое состояние (как запустить, что умеет).

Merge только когда оба ревью (auto и manual) пройдены и dev-сборка зелёная.

## 9. Риски и уточнения с пользователем до старта

Нужно прояснить **до** начала SCRUM-5, чтобы не переделывать:

1. **SCRUM-3** — что с ним делать? Закрыть / переформулировать / оставить как placeholder?
2. **SCRUM-5, семантика «редактирования»** — inline-edit по клику на текст или модалка с формой? По умолчанию предлагаю inline, как у большинства типовых todo.
3. **SCRUM-5, completed-флаг** — нужен чекбокс «выполнено» (стандартная семантика todo)? В тексте тикета сказано «самые распространённые виды тудушек» — трактую как «да, нужен», но требует подтверждения.
4. **SCRUM-5, поля todo** — только title или ещё description/dueDate/priority? Рекомендую MVP: id + title + completed + createdAt.
5. **SCRUM-6, критерии фильтрации** — по completed-статусу (`all / active / completed`), по тексту, по обоим? От ответа зависит сигнатура кастомного хука.
6. **SCRUM-7, направление сортировки** — только A→Z или toggle A↔Z? Учитывать ли регистр и локаль (`localeCompare`)?
7. **SCRUM-8, ключ и формат хранения** — фиксируем `todos:v1` для будущей миграции схемы? Делать ли debounce записи?
8. **Тесты** — нужны ли unit-тесты (Vitest + React Testing Library) в рамках упражнения, или хватает ручной проверки? Это влияет на acceptance criteria каждой спеки.
9. **Tooling** — Vite (рекомендую) vs CRA vs Next.js (избыточно для SPA без backend). Подтвердить Vite + React + TS + Tailwind v3 (или v4, если ок с новизной).
10. **Линтинг/форматирование** — ESLint + Prettier ставим сразу в SCRUM-5 или потом? Влияет на baseline auto review.
11. **CodeRabbit** — есть ли доступ/готовность подключить GitHub App? Если нет — обходимся `/ultrareview`.
12. **CLAUDE.md** — заводим сразу в SCRUM-5 (минимальный) или ждём Cartograph? Рекомендую минимальный сразу.
13. **GitHub remote** — подтвердить имя репозитория и владельца (URL `origin`). Если remote ещё не настроен — настроим перед первым push.
14. **CI вместе с SCRUM-5** — ставим `.github/workflows/ci.yml` в PR по SCRUM-5 (рекомендую) или отдельным PR после SCRUM-5?
